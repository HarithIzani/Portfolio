1. Project Overview

Portable Traffic Data Gathering Tool is an edge-deployed computer vision system for near real-time traffic monitoring under all environmental lighting conditions. Built as a Final Year Project and awarded the Gold Medal, the system uses a dual-model framework trained via transfer learning on the Berkeley DeepDrive dataset (BDD100K), deployed on a Raspberry Pi for portable field use.

The core innovation is automatic luminance-based model switching: a custom relative luminance function (based on the WCAG standard) determines whether the incoming video frame is a "day" or "night" scene, and routes it to the appropriate specialized TFLite model. This allows robust detection across bright daylight, dim dusk, and low-light nighttime conditions without manual intervention.

Key Capabilities

  • Dual-Model Switching — Automatic day/night model selection per frame using WCAG relative luminance
  • 11-Class Detection — Detects bicycle, bus, car, motorcycle, pedestrian, rider, train, truck, other vehicle, other person, trailer
  • Edge Deployment — Runs entirely on Raspberry Pi using TFLite runtime (no cloud dependency)
  • Near Real-Time Processing — TFLite quantized models optimized for ARM CPU inference
  • Automated Data Recording — Per-frame master record exported to CSV with timestamps, luminance, model used, and per-class counts
  • Transfer Learning — Fine-tuned from SSD MobileNet V2 FPNLite 640 pretrained checkpoint
  • Adaptive Confidence Thresholds — Day model uses 0.5 threshold; night model uses 0.3 for greater sensitivity in low light

2. Architecture & System Flowchart

The system follows a linear yet branching architecture: a single video source is read frame-by-frame, with each frame independently evaluated for luminance and routed to either the Day or Night TFLite model. Both paths converge back into a shared recording and display loop. On exit, all accumulated data is flushed to a CSV file.

Luminance-Based Model Switching

The luminance function is derived from the WCAG 2.1 relative luminance standard, which weights the RGB channels by their perceptual contributions to human brightness perception:

def luminance(img):
    avg_color_per_row = np.average(img, axis=0)
    BGR = np.average(avg_color_per_row, axis=0)
    return (0.2126 * BGR[2]) + (0.7152 * BGR[1]) + (0.0722 * BGR[0])

This is applied to the resized input frame (already in BGR format from OpenCV). The result is compared against the empirically calibrated threshold of 84, determined by sampling the Berkeley DeepDrive test set:

ConditionMeasured Lumen RangeExample Values
Bright daytime~91–13491, 98, 92, 134, 92, 116, 116, 116, 111, 102 → avg 107
Dim / dusk~67–10791, 85, 72, 78, 107, 90, 96, 80, 67, 75 → avg 84
Dark / nighttime~13–7416, 13, 74, 22, 16, 44, 31, 56, 50, 53 → avg 37
Design Decision
The threshold of 84 corresponds to the average luminance of "dim" conditions — the boundary between scenes where the day model starts degrading and the night model begins to outperform it. This was calibrated empirically from the BDD100K test subset.

5. Detection Pipeline (Detailed)

The pipeline executes inside a while(video.isOpened()) loop. Each iteration processes one frame through 7 logical steps. Both the Day and Night branches are structurally identical — only the model interpreter and confidence threshold differ.

Step 1 — Initialization & Model Loading

fyp.py lines 1–51 Startup

All static resources are loaded once at startup before the main loop begins. Two separate TFLite interpreters are instantiated and their tensors pre-allocated, eliminating per-frame allocation overhead.

# Both models loaded at startup
interpreter_day = Interpreter(MODEL_DAY)
interpreter_day.allocate_tensors()
interpreter_night = Interpreter(MODEL_NIGHT)
interpreter_night.allocate_tensors()

# Input tensor dimensions extracted from day model
# (both models share the same 640×640 input shape)
height = input_details_day[0]['shape'][1]   # 640
width  = input_details_day[0]['shape'][2]   # 640

The boxes_idx, classes_idx, scores_idx offsets are set to 1, 3, 0 to match the TF2 model output ordering used during export.

Step 2 — Video Capture & Frame Preprocessing

fyp.py lines 53–103 Per Frame

Each frame is read via video.read(). If the source is exhausted, the loop exits cleanly. The frame undergoes two transformations: one for inference (resized to model input, BGR→RGB), and one for display (resized to 800×470 for the Raspberry Pi screen).

ret, frame = video.read()
if not ret: break

# For inference: RGB + resize to model input (640×640)
frame_rgb     = cv2.cvtColor(frame, cv2.COLOR_BGR2RGB)
frame_resized = cv2.resize(frame_rgb, (width, height))
input_frame   = np.expand_dims(frame_resized, axis=0)

# For display: resize to RPi screen (800×470)
frame_show = cv2.resize(frame, (800, 470))
Configuration Note
The source path SOURCE must be updated before deployment. The default value is showcase9min10f.mp4. For live camera input, replace with cv2.VideoCapture(0).

Step 3 — Luminance Calculation & Model Selection

fyp.py lines 105–107 Core Logic

The WCAG relative luminance function is applied to frame_resized (the 640×640 inference input). This produces a single float lumen that determines which model branch executes for this frame.

lumen = luminance(frame_resized)

if lumen >= 84:
    # → Day model branch (lines 108–167)
if lumen < 84:
    # → Night model branch (lines 169–229)
Note on Branch Logic
Both branches are evaluated using separate if statements (not if/elif). Since the conditions are mutually exclusive (≥84 vs <84), only one executes per frame. This is functionally equivalent to an if/else structure.

Step 4 — Model Inference

fyp.py lines 108–118 (Day) / 170–180 (Night) Day Night

The selected frame is normalized and fed to the appropriate TFLite interpreter. Normalization scales pixel values from [0, 255] to approximately [−1, 1] using the standard MobileNet normalization parameters.

# Normalize pixel values (both models are floating point, non-quantized)
input_frame = (np.float32(input_frame) - 127.5) / 127.5

# Run inference on selected model
interpreter_day.set_tensor(input_details_day[0]['index'], input_frame)
interpreter_day.invoke()

# Retrieve outputs
boxes   = interpreter_day.get_tensor(output_details_day[boxes_idx]['index'])[0]
classes = interpreter_day.get_tensor(output_details_day[classes_idx]['index'])[0]
scores  = interpreter_day.get_tensor(output_details_day[scores_idx]['index'])[0]

The three output tensors are: boxes (normalized [ymin, xmin, ymax, xmax] coordinates), classes (integer class indices), and scores (confidence values 0–1).

Step 5 — Result Filtering & Master Record Update

fyp.py lines 120–144 (Day) / 182–206 (Night) Data Recording

Detections above the confidence threshold are counted per class. A temporary classes_dictionary is built for this frame, then appended to the master_record dict along with timestamp, luminance value, and model name.

classes_dictionary = {cls: 0 for cls in labels}  # reset per frame

for i, score in enumerate(scores):
    if scores[i] >= 0.5:   # (0.3 for night)
        classes_dictionary[labels[int(classes[i])]] += 1

# Record this frame's results
master_record['DateTime'].append(datetime.now().strftime("%d/%m/%Y %H:%M:%S"))
master_record['Lumin'].append(lumen)
master_record['Model'].append('Day')   # or 'Night'
for category in classes_dictionary:
    master_record[category].append(classes_dictionary[category])

Step 6 — Bounding Box Rendering & Display

fyp.py lines 146–167 (Day) / 208–229 (Night) Display

All detections above the confidence threshold are drawn on frame_show (the 800×470 display frame). Normalized bounding box coordinates are remapped to display pixel coordinates using imH and imW, with max() and min() clamping to prevent out-of-bounds draws. Each box is labelled with the class name and confidence percentage.

for i in range(len(scores)):
    if 0.5 <= scores[i] <= 1.0:
        ymin = int(max(1, boxes[i][0] * imH))
        xmin = int(max(1, boxes[i][1] * imW))
        ymax = int(min(imH, boxes[i][2] * imH))
        xmax = int(min(imW, boxes[i][3] * imW))

        cv2.rectangle(frame_show, (xmin, ymin), (xmax, ymax), (10, 255, 0), 4)

        label = '%s: %d%%' % (labels[int(classes[i])], int(scores[i]*100))
        cv2.putText(frame_show, label, (xmin, label_ymin-7),
                    cv2.FONT_HERSHEY_SIMPLEX, 0.7, (0, 0, 0), 2)

# HUD overlay showing active model and luminance
cv2.putText(frame_show, 'Model: Day  Lumen: {:.2f}'.format(lumen), (10, 50),
            cv2.FONT_HERSHEY_SIMPLEX, 0.7, (10, 255, 0), 2)

cv2.imshow('RaspPi Detector', frame_show)

Step 7 — CSV Export on Exit

fyp.py lines 237–243 Output

After the main loop exits (source exhausted or user presses q), the video capture is released and the master_record dictionary is converted to a Pandas DataFrame and exported to a CSV file at a hardcoded path on the Raspberry Pi desktop.

video.release()
cv2.destroyAllWindows()

master_record_conv = pd.DataFrame.from_dict(master_record)
master_record_conv.to_csv('/home/fyp/Desktop/Master_Record.csv')
Deployment Note
The CSV output path /home/fyp/Desktop/Master_Record.csv is hardcoded for the Raspberry Pi deployment user fyp. This must be changed if deploying under a different username or on a non-Linux system.

6. Transfer Learning & Training Pipeline

Both models were produced by fine-tuning the SSD MobileNet V2 FPNLite 640 pretrained checkpoint from the TensorFlow Model Zoo using the TF Object Detection API. The training pipeline processed annotated frames sourced from the Berkeley DeepDrive (BDD100K) dataset.

Training Workflow

1
Dataset Acquisition BDD100K
Berkeley DeepDrive (BDD100K) video frames covering diverse traffic scenarios under varied weather, time-of-day, and road conditions. Frames relevant to 11 target classes were extracted and prepared as JPG images.
2
Annotation with LabelImg XML
Each image was annotated using LabelImg, producing Pascal VOC XML files with bounding boxes for each of the 11 target classes. The xml_edit.py utility then cleaned annotations by removing traffic sign, traffic light, and none classes that were outside the project scope, and normalized filename fields across the dataset.
3
TFRecord Generation TFRecord
The generate_tfrecord.py script converts the XML + JPG pairs into binary TFRecord files, the native format for the TF Object Detection API training pipeline. A label map (label_map.pbtxt) defines the integer-to-class-name mapping used during training.
4
Transfer Learning — Two Models Day Night
Two separate models were trained: one on day-condition frames and one on night-condition frames. Both were fine-tuned from the SSD MobileNet V2 FPNLite 640 pretrained weights. The day model converged at ckpt-11. Training was managed through the TF Object Detection API pipeline config (pipeline.config).
5
TFLite Conversion & Quantization Edge
The trained SavedModel checkpoints were exported and converted to TFLite format for edge deployment. The resulting .tflite files are float32 (non-quantized), preserving maximum accuracy on the Raspberry Pi ARM CPU while remaining lightweight enough for real-time inference.
6
Luminance Calibration Validation
The experimental notebook tested 1,000+ frames from the BDD100K test split to determine the luminance range boundaries. The threshold of 84 was selected as the inflection point where day and night model performance crossed over, based on measured average luminance values across "dim" condition images.

10. API Reference — fyp.py

Functions

luminance(img)

Location: fyp.py, lines 64–67

Computes the WCAG 2.1 relative luminance of an image frame. Used to determine whether each frame should be processed by the Day or Night model.

ParameterTypeDescription
imgnp.ndarrayBGR image array, shape [H, W, 3], as returned by OpenCV. Typically the 640×640 resized inference frame.
ReturnsTypeDescription
luminance scorefloatWeighted average luminance value. Compares to threshold 84: ≥84 → Day, <84 → Night.
def luminance(img):
    avg_color_per_row = np.average(img, axis=0)   # average across rows → [W, 3]
    BGR = np.average(avg_color_per_row, axis=0)   # average across cols → [3]
    # WCAG relative luminance: R=0.2126, G=0.7152, B=0.0722
    # Note: OpenCV uses BGR order, so BGR[2]=R, BGR[1]=G, BGR[0]=B
    return (0.2126 * BGR[2]) + (0.7152 * BGR[1]) + (0.0722 * BGR[0])
Implementation Note
This function operates on the raw pixel mean (not linearized gamma-corrected values as in the strict WCAG standard), which is intentional. The threshold was calibrated empirically to this simplified version. The comment in the code notes that img.mean() was also tested as a fallback.

xml_edit.py

Batch utility to clean and normalize LabelImg XML annotation files before TFRecord conversion. Performs two operations on every .xml file in a directory:

  • xmlDeleteObject — Removes all <object> entries where the class name is traffic sign, traffic light, or none
Path Configuration Required
The train_path variable at the top of xml_edit.py is hardcoded to the developer's local filesystem. Update this path before running.

14. Deployment Notes

Raspberry Pi Setup

  1. Install dependencies: pip install tflite-runtime opencv-python numpy pandas
  2. Copy the RaspberryPi Build/ directory to the Raspberry Pi (e.g. /home/fyp/fyp_project/)
  3. Extract TFLite_Model.zip into the same directory to produce TFLite_Model/detect_day.tflite and TFLite_Model/detect_night.tflite
  4. Update SOURCE in fyp.py to point to your video file, or change to cv2.VideoCapture(0) for live camera
  5. Optionally update the CSV output path at the bottom of the script
  6. Run: python3 fyp.py
  7. Press q to stop the session and export the CSV
Display Resolution
The script hardcodes an 800×470 display resolution optimized for the official Raspberry Pi 7" touchscreen. If using a different display, update imW and imH accordingly. The bounding box coordinate remapping uses these values, so incorrect dimensions will shift box positions.
Camera Input vs. Video File
Live camera input (cv2.VideoCapture(0)) gives true real-time monitoring but requires the commented-out lines for imW/imH to be re-enabled so that actual capture dimensions are used instead of the hardcoded 800×470 values. When using a video file, the hardcoded display dimensions work correctly for the Raspberry Pi screen.

Known Limitations

  • The luminance threshold (84) was calibrated on BDD100K test data. Performance under unusual lighting conditions (e.g. fog, tunnel lighting, infrared cameras) may require recalibration
  • Both models are float32 (non-quantized). Integer quantization could improve inference speed on Raspberry Pi at the cost of some accuracy
  • The master record dict grows unbounded in memory during a session. For very long deployments (multi-hour recordings), periodic CSV flush intervals should be considered
  • The traffic sign and traffic light classes were explicitly excluded from training — the system will not detect these objects even if present
