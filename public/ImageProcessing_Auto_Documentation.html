1. Project Overview

ImageProcessing Auto is a modular, automated document image processing pipeline built for high-volume camera-captured document digitization. It processes raw JPG images through up to 8 sequential steps — from ingestion and metadata extraction through AI-powered orientation correction, deskewing, cropping, separator detection, and batch organization.

Key Capabilities

  • GPU-Accelerated AI Inference — Orientation detection via NVIDIA Triton Inference Server with ONNX models
  • Multi-Core CPU Processing — Parallel deskewing, cropping, and separator detection using all available CPU cores
  • Tkinter GUI — Simple Start/Stop interface with real-time color-coded console output
  • Headless CLI Mode — Run the full pipeline without UI via run_orchestrator.py
  • Per-Image Tracking — Every image tracked through all stages via JSON files
  • Configurable Pipeline — Enable/disable individual steps, tune Triton parameters, set folder paths
  • GPU Auto-Tuning — Detects GPU memory and automatically optimizes batch sizes and Triton instances

7. Pipeline Steps (Detailed)

All steps are loaded dynamically by the PipelineOrchestrator from pipelines/ using importlib. Each step is a class with a run() or execute() method. Steps run sequentially; the pipeline halts on the first failure (fail-fast). Each step can be independently enabled/disabled via config.

Step 01: Ingestion

File: pipelines/01_ingestion.py · Class: Ingestion · 368 lines · CPU

Purpose: Copy raw images from the input folder to a backup location and initialize per-image tracking JSON files.

Algorithm

  1. Scan input folder for supported formats: .jpg, .jpeg, .png, .tif, .tiff
  2. Skip images that already have a tracking JSON (idempotent re-runs)
  3. For each new image (parallel via ThreadPoolExecutor):
    • Copy file to backup folder
    • Extract metadata: dimensions, DPI, color mode, file hash (MD5)
    • Create tracking JSON with ingestion stage data
  4. Generate ingestion_manifest.json in state folder

Inputs & Outputs

InputRaw images from folders.input
OutputBackup copies in folders.backup + tracking JSONs in folders.tracking
Manifeststate/ingestion_manifest.json

Parallelism

ThreadPoolExecutor with max_workers = config value (0 = all CPU cores).

Step 03: Orientation Predict

File: pipelines/03_orientation_predict.py · Class: OrientationPredict · 448 lines · GPU (Triton)

Purpose: Classify image orientation (0°, 90°, 180°, 270°) using a ViT-based ONNX model served by Triton Inference Server.

Preprocessing Pipeline (must match model training)

  1. Decode image as RGB (PIL)
  2. Resize shorter side to 256px (bicubic, maintain aspect ratio)
  3. Center crop to 224×224
  4. Scale pixel values to [0, 1]
  5. Convert HWC → CHW layout
  6. Normalize with ImageNet statistics: mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]

Inference Strategy — High Throughput

  1. Preprocess all images upfront using a wide thread pool (up to 64 workers) to eliminate per-batch CPU overhead
  2. Stack into batches of configured batch size (default: 64)
  3. Submit batches concurrently via ThreadPoolExecutor (max 8 in-flight) to keep Triton's GPU saturated
  4. Collect results as futures complete — each batch returns class predictions with softmax confidence

Triton Model Details

Modeldoc_ori
Inputx — FP32 [batch, 3, 224, 224]
Outputlogits — FP32 [batch, 4]
Classes0° (index 0), 90° (index 1), 180° (index 2), 270° (index 3)
ProtocolgRPC on configured triton.server_url

Outputs

Predictions JSONstate/triton_predictions/03_orientation_predictions_{timestamp}.json

Step 04: Orientation Rotate

File: pipelines/04_orientation_rotate.py · Class: OrientationRotate · 296 lines · CPU

Purpose: Apply the rotation detected in Step 03 to physically rotate images to upright orientation.

Algorithm

  1. Load the most recent 03_orientation_predictions_*.json from state/triton_predictions/
  2. Skip images classified as 0° (already upright)
  3. For each rotated image (90°, 180°, 270°), in parallel via ThreadPoolExecutor:
    • Read image with cv2.imread()
    • Apply cv2.rotate() — pixel-perfect rotation (no interpolation artifacts, no pixel loss)
    • Overwrite original file in-place with cv2.imwrite() at JPEG quality 100
  4. Update tracking JSONs with rotation status

Rotation Mapping

PredictedOpenCV FunctionEffect
0°None (skip)Already upright
90°cv2.ROTATE_90_COUNTERCLOCKWISERotate 90° CCW
180°cv2.ROTATE_180Rotate 180°
270°cv2.ROTATE_90_CLOCKWISERotate 90° CW
ℹ️ Note
Images are modified in-place in the input folder. Originals are preserved in the backup folder from Step 01.

Step 05: Deskew

File: pipelines/05_deskew.py · Class: Deskew · 332 lines · Worker: deskew_worker.py · CPU

Purpose: Detect skew angle of each document image using contour analysis.

Algorithm (per image, in deskew_worker.py)

  1. Read image with OpenCV
  2. Convert to grayscale
  3. Apply Gaussian blur (3×3 kernel)
  4. Otsu thresholding (automatic)
  5. Morphological close (25×25 kernel) — fills gaps to form document contour
  6. Find external contours
  7. Get largest contour → cv2.minAreaRect()
  8. Extract rotation angle, normalize to [-45°, 45°]
  9. Store angle in tracking JSON
⚠️ Important
This step detects but does NOT apply rotation. The actual rotation is deferred to the crop step (Step 06) which performs atomic rotate+crop. The detected angle is stored in tracking only.

Parameters

ParameterValueNotes
Rotation Threshold0.5°Only flag if angle exceeds this
Gaussian Kernel3×3
Morphological Kernel25×25Legacy size matching original app
JPEG Quality100Maximum quality
InterpolationLANCZOS4Highest quality interpolation

Parallelism

Windows: ThreadPoolExecutor (avoids multiprocessing bootstrap issues). Unix/Linux: ProcessPoolExecutor (better for CPU-bound work).

Step 06: Crop

File: pipelines/06_crop.py · Class: CropStep · 215 lines · Worker: crop_worker.py · CPU

Purpose: Crop document images by detecting the document boundary and extracting it with simultaneous rotation correction (atomic deskew + crop).

Method A Algorithm (for all images, in crop_worker.py)

  1. Read image with OpenCV
  2. Convert to grayscale
  3. Gaussian blur (3×3)
  4. Otsu thresholding
  5. Morphological close (5×5 kernel)
  6. Find largest external contour
  7. cv2.minAreaRect() → get rotated bounding box
  8. cv2.getRotationMatrix2D() → rotation matrix
  9. cv2.warpAffine() with LANCZOS4 + BORDER_REPLICATE → precise rotation
  10. cv2.getRectSubPix() → extract cropped region (sub-pixel precision)
  11. Save overwriting original at JPEG quality 100
ℹ️ Note
Method A uses Otsu (automatic threshold) for normal/light documents. A Method C also exists in crop_worker.py using HSV Value channel with a fixed threshold of V≥53, designed for dark documents — but Method A is applied to all images in the current pipeline.

Entry Method

Unlike other steps, CropStep uses execute() instead of run(). The orchestrator handles both method names.

Step 07: Separator Detect

File: pipelines/07_separator_detect.py · Class: SeparatorDetect · 423 lines · Worker: separator_detect_worker.py · CPU

Purpose: Classify each image as a "Separator" (black divider sheet) or "Normal" (document page) using a fast brightness-based algorithm. No GPU needed.

P99 Brightness Algorithm

  1. Open image with PIL
  2. Convert to grayscale, resize to 128×128 (uses JPEG draft mode for speed)
  3. Calculate P99 — the 99th percentile of pixel brightness values
  4. Classification:
    • P99 < 50 → Separator (near-black page)
    • P99 ≥ 50 → Normal document (contains bright paper)
  5. Calculate confidence score based on distance from threshold

Confidence Scoring

  • Separator: confidence = max(0, 1 - p99/50) — higher confidence for darker pages
  • Normal: confidence = min(1, (p99 - 50) / (255 - 50)) — higher confidence for brighter pages

Step 08: Batch Separation

File: pipelines/08_batch_separation.py · Class: BatchSeparation · 525 lines · CPU

Purpose: Split the processed images into numbered batch folders based on separator positions detected in Step 07, and move separator images to a dedicated folder.

Algorithm (6 phases)

  1. Load images — Get all JPGs from input folder, sorted naturally
  2. Load CSV metadata — Single-pass read of all CSV files from input folder (O(1) lookups later)
  3. Classify from tracking — Read Step 07 classification from each image's tracking JSON
  4. Create batch groups — Split image list at separator positions; images between consecutive separators form a batch; if no separators, all images form one batch
  5. Move separators — Move all separator images to folders.separator
  6. Move batches (parallel) — Using ThreadPoolExecutor (8 threads):
    • Determine next batch number from existing folders
    • Create folder named {NNNNNN}-{BoxName} (e.g., 000001-Estate_Box_5)
    • Move batch images using shutil.move()

Batch Folder Naming

D:/batches/
├── 000001-Estate_Box_5/
│   ├── SAX10_IMG_20250805_100438.jpg
│   ├── SAX10_IMG_20250805_100439.jpg
│   └── ...
├── 000002-Estate_Box_6/
│   └── ...
└── ...

Box name is derived from the first image's CSV metadata (box_details field). Invalid filename characters are sanitized.

⚠️ Important
This step moves files (not copies) — images are removed from the input folder. The backup from Step 01 is the safety net.

8. Worker Modules

CPU-heavy operations are isolated into separate worker modules for proper threading/multiprocessing serialization (pickle compatibility).

8.1 deskew_worker.py (144 lines)

Function: _process_image_worker(args: Tuple) → Dict

Takes a tuple of (image_path, rotation_threshold, gauss_kernel, morph_kernel, jpeg_quality, interpolation). Returns a dict with filename, status, message, rotation_angle, processing_time.

Includes Unix-only SIGALRM timeout (30 seconds) — gracefully skipped on Windows.

8.2 crop_worker.py (177 lines)

Two functions:

  • _process_method_a(args) — Otsu + Grayscale + 5×5 morphological close (for normal docs)
  • _process_method_c(args) — HSV Value channel + fixed threshold 53 + 25×25 close (for dark docs)

Both share _crop_and_deskew() — the core rotate+crop function using warpAffine + getRectSubPix.

8.3 separator_detect_worker.py (70 lines)

Function: _process_image_worker(args: Tuple) → Dict

Lightweight P99 brightness calculation using PIL at 128×128 resolution. JPEG draft mode for speed.

9. Triton Integration

9.1 Overview

NVIDIA Triton Inference Server runs as a Docker container serving ONNX models via gRPC. The application connects as a client for GPU-accelerated inference.

9.2 triton_api.py (251 lines) — ACTIVE

Synchronous gRPC client wrapper (TritonAPI class):

  • classify(model, images, ...) — Single-batch classification with preprocessing
  • classify_batch_async(model, batch, images, ...) — Pre-processed batch inference (used by Step 03 in thread pool for concurrent submission)
  • detect(model, images, ...) — Generic detection interface

Handles softmax conversion of logits to probabilities and argmax class selection.

9.3 triton_classifier_client.py (387 lines) — UNUSED

Alternative async gRPC client using tritonclient.grpc.aio. Provides higher-throughput async batching but is not referenced by any active pipeline step.

9.4 serve_models.py (234 lines)

Docker launcher for Triton:

  1. Verifies models exist and Docker is running
  2. Attempts GPU mode first (--gpus all)
  3. Falls back to CPU-only if GPU fails
  4. Exposes ports: HTTP 8000, gRPC 8001, Metrics 8002
  5. Uses Docker image: nvcr.io/nvidia/tritonserver:24.12-py3

9.5 Docker Command (GPU mode)

docker run --rm --gpus all \
  -p 8000:8000 -p 8001:8001 -p 8002:8002 \
  -v {models_dir}:/models \
  --name triton-server \
  nvcr.io/nvidia/tritonserver:24.12-py3 \
  tritonserver --model-repository=/models \
  --grpc-port=8001 --http-port=8000 --metrics-port=8002

02
Generate Data CPU
Operator.xlsx + Metadata.xlsx → enrich tracking JSON  ·  Generate transfer_{timestamp}.csv
03
Orientation Predict GPU — Triton
Preprocess → batch gRPC inference → classify 0°/90°/180°/270°  ·  Save predictions to state/triton_predictions/
04
Orientation Rotate CPU
Load predictions → rotate non-upright images IN-PLACE via cv2.rotate()  ·  Input images now physically rotated
05
Deskew CPU
Detect skew angle only → store angle in tracking JSON  ·  No rotation applied (deferred to crop step)
06
Crop CPU
Otsu → minAreaRect → warpAffine → getRectSubPix  ·  Atomic rotate+crop IN-PLACE  ·  Images now deskewed and cropped
07
Separator Detect CPU
128×128 grayscale → P99 percentile  ·  p99 < 50 → "separator"  |  p99 ≥ 50 → "normal"  ·  Save predictions
08
Batch Separation CPU
Split at separator positions → numbered batch folders  ·  Move docs → batches/NNNNNN-BoxName/  ·  Move seps → separators/  ·  Input folder now empty
Final Output: batches/NNNNNN-BoxName/ (organized documents)  ·  separators/ (black sheets)  ·  backup/ (original copies)  ·  tracking/*.json (full processing history)
